## SASS — Syntactically Awesome Style Sheets

The reason SASS exists is to automate the creation of regular cascading style sheets (CSS) by adding powerful features missing from CSS itself. SASS is a so-called *CSS pre-processor*, and is the defector winner in the category, having out-lasted the other early contenders in the space,  most notably LESS (*Leaner Style Sheets*).

The process is simply:

1. Write your style sheets in SASS
2. Compile the SASS stylesheet to CSS
3. Import the compiled CSS into your HTML files using a `<link rel="stylesheet">` tag in `<head>`, or an  `@import`  at-rule in a `<style>` tag.

For historical reasons the SASS pre-processor accepts two different syntaxes, the original indent-based SASS syntax, and the newer curly-braces-based SCSS syntax. The newer SCSS syntax is a super-set of regular CSS, so developers generally find it easier to learn, and because it uses curly braces to group statements rather than relying purely on indentation, it is more robust when copy-and-pasted. **We will only be using the newer SCSS syntax**!

Yes, this is very confusing, we are going to be using a piece of software named SASS that supports two syntaxes, one of which is named the *SASS syntax* and saved in `.sass` files, and to the other is named the *SCSS syntax* and saved in `.scss` files, but both of which are referred to as *SASS files*, and both can be converted to regular CSS with the `sass` command!

To keep things as un-unclear as possible, we're going to utterly ignore the older SASS syntax and only use the newer SCSS syntax, and we're going to **refer to our SASS code in that SCSS syntax as simply _'SASS'_**, like just about the entire internet does these days!

## SASS Basics

Firstly, because we are using the newer syntax, all our code will be saved in files with the `.scss` file extension.

Secondly, because SASS is a super-set of CSS, **all valid CSS code is valid SASS code**!

In addition to being able to do everything CSS can do SASS adds many features that CSS does not support. We're not going to explore anywhere near all of them. Instead, we're going to focus on the ones we're mostly likely to use when working with static site generators like Jekyll, and especially when working with Bootstrap inside Jekyll.

Here's a quick overview of the features we'll be exploring:

* **Selector nesting** — using curly braces to control the targeting of style declarations more clearly
* **SASS Variables** — both much simpler to use and more powerful than CSS custom properties
* **Conditionals and Loops** — implemented as SASS-only at-rules, things like `@if`, `@each` etc..
* **SASS Operators** — like CSS functions, but SASS-only
* **Inheritance** — import the styles from one selector into another to avoid code duplication
* **Mixins** — SASS's implantation of style templates

### The Atoms of a SASS Style Sheet

Because SASS is a super-set of CSS, the pieces used to assemble as SASS stylesheet will look familiar:

1. **Style Declarations** — definitions for *style properties* applied to different parts of an HTML document *selectors*
2. **Style Properties** — every HTML element on a page has a list of specific pieces of relevant style information associated with it, these are the properties CSS can alter. For example, the element's colour, the type face to be used any text within the element, how much padding there should be around the element, and so on.
3. **Selectors** — a syntax for expressing the set of elements style declarations apply to, e.g. `p` applies declarations to all paragraphs, `#waffle` applies declarations to just the element with the ID `waffles`, and `p.pancakes` applies declarations to all paragraphs with the *class* `pancakes`.
4. **Custom CSS Properties** — also known as *CSS Variables* are associated with HTML elements in the same way regular style properties are, but their value does not directly affect the page's presentation, they need to be referenced in regular style declarations to take an effect (see [instalment 183](./pbs183) for the details).
5. **SASS Variables** — traditional variables that exist in SASS and get rendered to the output CSS as their raw value.
6. **CSS at-rules** — CSS statements starting with an `@` that to perform special actions (rather than simply defining a style declaration), e.g. `@property` for defining custom CSS properties.
7. **SASS at-rules** — special SASS statements that look like CSS at-rules but do not actually exist in CSS and provide SASS features that do not exist in, e.g. `@if` & `@while`
8. **CSS Functions** — named computations and transformations that take arguments and convert them to values that can be used in style declarations.
9. **SASS Operators** — additional functions that only exist in SASS but share the same syntax as CSS functions, they are computed at compile-time rather than at run-time (like regular CSS functions).
10. **Mixin definitions** — SASS templates for generating CSS style declarations.

### Data Types in SASS

Again, because SASS is a super-set of CSS, all valid CSS values are valid SASS values.

Since SASS exists to generate CSS, only values that are valid in CSS can be outputted by SASS, but, it can use additional data types internally. Most notably, SASS support both lists and dictionaries.

The official SASS jargon refers to what we call lists as *Lists*, which is very convenient, but it refers to what we call dictionaries as *Maps*.

### Comments — SASS+CSS or SASS-Only

SASS supports two types of comments, and they have different meanings.

When use standard CSS comments (`/* … */`) in SASS files they get written into the generated CSS.

But when you use SASS comments (`// …`) they **only** exist in SASS, they are omitted from the generated CSS.

```scss
/* This is a comment in both SASS and CSS, and it will appear in the generated CSS */
/* Use comments like this to annotate the generated CSS */

// This on the other hand is a SASS comments and will not appear in the geneated CSS
// Use comments like this to describe SASS-specific aspects of your code
```

## Installing the SASS Command Line App

Many static site generators like Jekyll include SASS compilation libraries into their own code, allowing them to handle the SASS compilation for you. But to experiment with SASS without leveraging other tools we need to install the SASS command into command prompt. There are versions for all major operating systems, and you can find the full list on [the official install page](https://sass-lang.com/install/).

For Mac users like myself and Allison the simplest solution is [Homebrew](https://brew.sh):

```sh
# not a typo, the namespace really is nested like this!
brew install sass/sass/sass
```

Once you have `sass` installed you can compile SCSS files to CSS files with commands of the form:

```sh
sass source_file.scss output_file.css
```
