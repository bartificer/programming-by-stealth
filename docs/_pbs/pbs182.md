## Reminder — CSS Basics

CSS style sheets define how HTML content should be rendered — the sizes of things, the typography, the colours, the arranged and spacing of the various elements on the page, the borders to be added to the various elements, and so on.

The fundamental building blocks of CSS are:

1. **Style Declarations** — definitions for *style properties* applied to different parts of the page using *selectors*
2. **Style Properties** — every HTML element on a page has a list of specific pieces of relevant style information associated with it, these are the properties CSS can alter. For example, the element's colour, the type face to be used any text within the element, how much padding there should be around the element, and so on.
3. **Selectors** — a syntax for expressing the set of elements style declarations apply to, e.g. `p` applies declarations to all paragraphs, `#waffle` applies declarations to just the element with the ID `waffles`, and `p.pancakes` applies declarations to all paragraphs with the *class* `pancakes`.

These building blocks are applied to HTML pages following two fundamental principles:

1. Unless explicitly blocked, **style declarations cascade** into nested HTML elements.
2. Given how declarations cascade and how CSS is used, it's inevitable that there will be competing possible values for many style properties — there are well defined rules of **precedence** for CSS selectors, and this are used to resolve conflicts and assign a single specific values to each style property in each element.

## The Problem to be Solved

The feature colloquially referred as *CSS variables* was added to CSS to solve a common problem that we have been side-stepping by using Bootstrap's default style for all our web apps. We have been relying on Bootstrap CSS classes for ensuring consistency of colours, typography, spacing, and so on. We have not need to care about how using Boostrap's various `*-success` CSS classes ensure the same shade of green is always used for thing that succeed, or the `*-warning` classes for all types of warning messages, and so on. But were we to have been developing our own custom styles for each app we would have face this problem many times.

For example, were we to have chosen to use the Green from the Bartificer Creations logo for success we would have had tens or perhaps even hundreds of style decorations that align `#TODO` to various selectors. That's a meaningless number, making the CSS statements difficult to debug! Worse still, experimenting with different colour pallets would be very tedious because we would need to find-and-replace one meaningless value with another all over our code.

Clearly, there is a need for **named, re-usable values** within CSS style declarations!

## How CSS Implements 'Variables' — CSS Custom Properties

CSS is not a programming language in the traditional sense. You do not execute CSS code to achieve tasks, it's a kind of markup language for defining how HTML content should look, so it's not surprising that *variables* in CSS are not really variables at all.

The way CSS implements the concept of custom names values is by allowing users to invent their own style properties and assign values for those properties to elements using standard CSS selectors. These values just exist and have a name, but do not directly affect the look of the page. This implementation explains why these custom name-value pairs are not officially defined as *variables* in the documentation, but as *Custom Properties*.

This distinction matters, because unlike traditional variables, custom CSS properties don't have a single value within a document, their value can vary from element to element in the same way different parts of a page can have different colours because their matching elements have different values for the `color` property.

## Using CSS Custom Properties

The CSS specification actually defines two related mechanisms for defining custom properties, a simpler one that's sufficient for many situations, and a more complex but powerful optional additional mechanism.

### Custom CSS Property Names

Like with traditional variables, there are rules around how CSS properties can be named.

The rules are simply:

1. Must start with two dashes, i.e. `--`
2. Must follow all the same rules as traditional CSS property names like `background-color`.

In effect that means dash-delimited names of the form `--my-variable-name`, and they are case-sensitive.

By convention, custom properties are named following the same style as standard CSS style properties, so full words in all lower case separated with dashes.

### Assigning and Using Custom CSS Property Values

Other than their funny names, you assign values to custom properties just like you would for the traditional CSS properties, using style declarations.

To set the `color` property to `DarkBlue` for all top-level headers you would use:

```css
h1 {
  color: DarkBlue;
}
```

Similarly, you can set the value for the custom property `--pbs-logo-color` to `#00408d` on every top-level heading you would use:

```css
h1 {
  --pbs-logo-color: #00408d;
}
```

If you choose to do this, the custom property `--pbs-logo-color` will **only** have a value on `<h1>` tags and any elements inside `<h1>` tags. In other words, the value gets directly assigned on all `<h1>` tags and then *cascades* down into any tags within `<h1>` tags.

There are times you want to make these kind of fine-grain *'variable'* assignments, but that's not the most common problem-to-be-solved. What you're much more likely to want to do is assign a value at the document level, so that one definition cascades into each and every element on the page.

CSS defines a special selector that represents the document as a whole, `:root`, so to assign the value `#00408d` to the custom property `--pbs-logo-color` on all elements we would use:

```css
:root {
  --pbs-logo-color: #00408d;
}
```

We now have a named value of our choosing available within every element, but by definition, custom properties **have no effect on the rendering of an element**. At first glance that might sound like custom properties are useless, since they don't affect the style, but that's obviously not the case.

Drawing on our experience with traditional variables, what we've done here is analogous to declaring a Javascript variable but never using it. Just like `x=1;` doesn't really **do** anything until we use the variable's value  within another statement, css custom properties don't affect the page until their value is referenced within a style declaration.

To access the value of a custom property we need to use another advanced CSS feature we've been ignoring up to this point — a CSS function! we'll look at functions in a little more detail later, but for now we just need to know that custom property values are access with the `var()` function. You can call `var()` to fetch a value just about anywhere you can use a literal value. For example, we can use the colour stored in our custom property as a text colour with the style declaration `color: var(--pbs-logo-color)`. You must always provide a custom property name as the first argument to `var()`, but you can also optionally pass a fallback value as a second argument. If the custom properly is undefined, the fallback value will be used. For example, the following declaration would set the text colour to the value of our custom property, or if it's not defined, to black: `color: var(--pbs-logo-color, Black)`.

To see how to use the kind of *'global variable'* we've just defined, let's extend our code to use the value we stored in the `--pbs-logo-color` custom property in selectors to:

1. Set the colour of all headings to the logo colour
2. Set the background of all buttons to the logo colour, and the text on all buttons to white

Here's the updated CSS code:

```css
/* set the PBS logo color for the entire document */
:root {
  --pbs-logo-color: #00408d;
}

/* set the colour of all headings to the logo colour */
h1, h2, h3, h4, h5, h6{
  color: var(--pbs-logo-color);
}

/* set text colour of buttons to white, and the background colour to the logo colour */
button {
  color: white;
  background-color: var(--pbs-logo-color);
}
```

> **Pitfall** — custom properties need to be defined before they are used, so place your style definitions at the top of your style sheets.
{: .warning}

> **Pitfall** — setting a custom property to a new value for a sub-set of the document has no effect unless the variable is also **used** in a style declaration with a matching selector, and even then, the value will only be visible when that declaration has the highest specificity. This feels counter-intuitive, but actually follows the standard rules for all CSS declarations.
{: .warning}

### Advanced Custom Property Definitions with `@property` — Optional

For simple variable use you don't need `@property` definitions, but they are useful to know about, and you may prefer their more explicit approach. Personally, I prefer to use them, and There is another reason we care about them — Bootstrap 5 uses this approach for defining the custom properties it provides to facilitate customisations.

The syntax for `@property` definitions will look a little unusual because it is an example of another advanced CSS feature we've been ignoring so far — *at-rules*.

Unlike style definitions, at-rules don't work on the element level, they are global within the document. They exist to provide global controls. While we haven't used them explicitly, we have used them implicitly via Bootstrap — the `@media` at-rule is how the size-classes for the responsive behaviours are implemented.

If you want to use web fonts like those offered by [Google Fonts](https://fonts.google.com/) you can simplify the import using the `@import` at-rule. We'll see this in action in our final worked example. 

Let's get back to defining custom properties with `@property`. The syntax takes the following form:

```css
@property --my-property-name {
  syntax: "A VALID TYPE DEFINITION";
  inherits: true; /* or false */
  initial-value: "SOME VALUE"; /* optional */
}
```

The `initial-value` property acts as a default value, ensuring there at least **a** value for the custom property in every element.

> **Beware** because at-rules are not regular style declarations, functions can't be used within them, so we can't calculate an initial value based on the value of another custom property while declaring it.
>
> This is not a problem, it just requires slightly less elegant code — we declare the property with an `@property` declaration, but we omit the optional `initial-value` property completely. Then, later in the style sheet we use a regular style declaration on the `:root` element to set a default value.
>
> You can see this technique in action in the worked example later.

The `inherits` property use used to define whether or not the property cascades. By default, almost all CSS properties cascade, including custom properties. If you want to prevent your property from cascading, you can use an `@property` definition with `inherits` set to `false`.

The `syntax` property allows you to define the valid types or types for the values assigned to your custom property. Any style directives assigning invalid values will then be ignored. You can use this explicit approach to stop accidental mistakes causing spooky action at a distance.

The full set of rules for the `syntax` property is surprisingly complex, allowing some really quite advanced usages, so we'll constrain ourselves to a simplified sub-set.

Firstly, there are three kinds of type specifiers:

1. `*` — a special type to accept any valid CSS value.
2. `<type_name>` — any value fitting into a given broad class of value types, e.g. `<color>`, `<length>`, `<number>`, `<integer>` & `<percentage>`.
3. `specialValue` — a specific valid CSS special value like `auto`.

Secondly, these kinds of types can be combined using three operators:

1. A **space-separated list** of values is denoted by appending a `+` to a type, for example, to accept a space-separated list of colours, use `<color>+`.
2. A **comma-separated list** of values is denoted by appending a `#` to a type, for example, to accept a comma-separated list of integers, use `<integer>#`.
3. The `|` symbol can be used to allow multiple kinds of values, like a boolean **or**, for example, to accept valid lengths or the keyword `auto`, use `<length> | auto`.

Finally, the full list of supported types can be found on [this Mozilla Developer Network page](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/At-rules/@property/syntax), but here are a few you're most likely to need:

* `<number>` for any valid number, e.g. `42`, `3.1415`, `-1`, or `-273.15`.
* `<integer>` for any valid whole number, e.g. `-38`, `0`, or `42`.
* `<length>` for just about any valid CSS length value in absolute absolute relative units, but excluding percentages, e.g. `42px` or `2rem`.
* `<length-percent>` the same as `<length>`, but including percentage values, e.g. `42px`, `2rem`, or `50%`.
* `<string>` for any string of text.
* `<color>` for any valid colour value, including colour names, e.g. `white`, `#080808`, or `#000`.
* `<image>` for any valid CSS value that can specify an image, generally that will be a URL.

Let's put all this together by re-writing our logo colour example using an `@property` definition:

```css
/* declare the logo colour variable and give it a default value */
@property --pbs-logo-color {
  syntax: "<color>";
  inherits: true;
  initial-value: #00408d;
}

/* set the colour of all headers to the logo colour */
h1, h2, h3, h4, h5, h6 {
  color: var(--pbs-logo-color);
}

/* set buttons to use the logo colour as their background, and to have white text */
button {
  color: white;
  background-color: var(--pbs-logo-color);
}
```

Note that setting an initial value with `@property` eradicates any need for fallback values because the custom property will always have a value, and fallbacks are only triggered when a custom property has no value.

### Experiment 1 — Set the Variable to a Different Value on Some Elements

CSS custom properties behave like regular properties, so just like we can set different colours and border widths on different elements using CSS selectors, we can do the same with the value of our `--pbs-logo-color` custom property. Also, since we have explicitly configured out property to cascade down into child elements (`inherits: true;`) , the value should change on both the elements we explicitly target with our selector, and all the elements within those.

We can prove this by adding a style declaration that changes the value of our custom property to the very striking `MediumVioletRed` HTML colour on all paragraphs:

```css
p {
  --pbs-logo-color: MediumVioletRed;
}
```

Adding this to the end of the `<style>` tag and refreshing the page should change the background of the button to this new colour. Why? Because our custom property has been assigned a new value on every `<p>` tag, and that value has cascaded down into the `<button>` tag which is inside a `<p>` tag.

### Experiment 2 — Set the Variable to an Invalid Value on Some Elements

Because our example tag uses an advanced `@property` definition, our custom property has been explicitly declared as only accepting colours. This means that if we try to set it to any other value, CSS will ignore that style declaration.

We can do that by editing our new selector to change the value for our custom property from `MediumVioletRed` to `42px`, which is definitely not a valid colour!

Refreshing the page shows that our invalid assignment was ignored, so the button remains dark blue. Why? Because it is behaving exactly as if our invalid assignment did not exist.

## Using Custom Properties with Functions for Computed Values (Optional)

Using custom properties directly is already powerful, but to really get the most from custom properties you need to combine them with CSS functions to compute suites of related values from a single custom properties.

CSS function are too broad and advanced a topic to explore in detail here, so we've going to limit ourselves to just the two functions you're most likely to need. If this little preview whets your appetite, you can export CSS's full set of functions in [this MDN article](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Values/Functions).

Before we explore out two examples, let\'s take a moment to look at their standard usage:

```css
function-name(optional arguments)
/* or */
function-name(optional, arguments)
```

Yup, some functions expect their arguments to be space-delimited, and some comma-delimited — just to be clear, it's the function author that chooses how arguments are accepted, users of the function don't get to decide which to use! This means you always need to check the documentation for functions to understand how you should pass the arguments.

Note that the one function we've seen already, `var()` is an example of a function that expects comma-delimited arguments.

The arguments themselves can be any valid CSS value in any unit, and, they can be output of other CSS function calls, which includes the  `var()` function, making the values of custom properties available for use with functions.

### Basic Arithmetic with `calc()`

There are lots of reasons you might want to use a master numeric value to derive other related values, but sizes and spacings are great examples. For example, to make your paddings and margins internally consistent and pleasingly proportioned you could define a single base spacing size, and then use pleasing multiples of this base value throughout your style sheet. If you later decide you want more or less empty space, you can simply edit master spacing size in once place and update all the spacings at once while retaining their ratios. Bootstrap takes this approach, all the padding and margin utility classes like `p3`, `m5` etc. are multiples of a single master value Bootstrap refers to as the `space`. To make Bootstrap feel more or less roomy, you simply edit this one variable and all the margins and paddings adapt.

The `calc()` function expects to be passed a list of space-delimited arguments that form a sequence of values and arithmetic operators that make mathematical sense. The values must be numeric, but they can be in any unit, and only the basic four arithmetic operators can be used, specifically `+`, `-`, `*`, and `/`. You can find [detailed documentation on MDN](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Values/calc).

As an example, let's look at a simple implementation of the concept of using ratios of a base spacing size for all paddings and spacings:

```css
/* set a base spacing value */
:root {
  --space: 1rem; /* the width of one em-dash at the document root's font size (root-em)*/
}

/* calculate margins and paddings for the the document */
body{
  margin: 0px;
  padding: 0px var(--space) 0px var(--space); /* top left bottom right*/
}
p {
  margin-top: 0px;
  margin-bottom: var(--space);
}
h1, h2, h3, h4, h5, h6 {
  margin-top: 0px;
  margin-bottom: calc(var(--space) * 0.5);
}
blockquote { /* same vertical margin as paragraphs, but double the side margin and with padding */
  margin: 0px calc(var(--space) * 2) var(--space) calc(var(--space) * 2); /* top left bottom right*/
  padding: calc(var(--space) * 1.5)
}
```

### Colour Mixing with `colour-mix()`

Have you ever noticed how many professional websites seem to have a whole suite of colours that are all derived from just a small handful of base colours? Take Bootstrap for example, it works off just eight master colours named `primary`, `secondary`, `info`, `success`, `warning`,  `danger`, `light`, and `dark`. All the colours you see in the various Bootstrap components are calculated from those base values. This makes is very simple to customise the colours in Bootstrap — you don't need to specify each different shade to represent the foreground, background, light, and disabled variants of the colours, you just specify the master colours, and Bootstrap calculates the rest.

If you understand colour theory you'll be pleased to discover that the `color-mix()` function can work in multiple colour spaces. You can find all the details in [this MDN document](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Values/color_value/color-mix). I don't understand colour theory well enough to use this function to its full effect, but I understand enough to generate different shades of the same colour using a very simple technique — mixing more or less of the master colour with white to generate a series of lighter shades.

For this approach to work I use the darkest shade I will need as my reference colour, then calculate the rest.

Here's a very simple example generating just one very light background colour from a master colour:

```css
/* set an accent colour — this will be the darkest shade used */
:root {
  --accent-color: #00408d; /* the PBS shade of dark blue */
}

/* Use shades of the accent colour for quotations */
blockqote {
  color: var(--accent-color);
  background-color: color-mix(in srgb, var(--accent-color) 10%, white); /* calculate a suitably light background shade */
}
```

2. Uses variables derived from other variables to reduce the number of variables the user needs to choose values for.

The final worked example can be found in the instalment ZIP as `pbs182-example2.html`. Open that file in your favourite browser to see the final result, and in your favourite code editor to review the code.

### The Page's Structure

To understand how the CSS works it's important to understand the structure of page's body. I've kept it simple enough to avoid confusion, but with just enough complexity to demonstrate the key concepts. 

The page has a top-level heading, paragraph, and block quote. The block quote contains nested paragraphs, one of which has a CSS class assigned, specifically, `class="author"`. This is the page's complete body:

```html
<h1>PBS 182 — Example 2 (CSS Custom Properties)</h1>

<p>True wisdom is a rare and wonderful thing, and it should be cherished where ever it's found.</p>

<blockquote>
  <p>"The world is a dangerous place to live; not because of the people who are evil, but because of the people who don't do anything about it."</p>
  <p class="author">— Albert Einstein</p>
</blockquote>
```

### The Style Sheet's Organisation

The style sheet follows best practice and is arrange in the following order:

1. The at-rules:
   1. An `@import` rule to import three Google Fonts ([these three to be specific](https://fonts.google.com/share?selection.family=Indie+Flower|Ubuntu+Mono:ital,wght@0,400;0,700;1,400;1,700|Ubuntu+Sans:ital,wght@0,100..800;1,100..800))
   2. The `@property` definitions for all custom properties
2. The root-level definitions for the calculated custom properties
3. The style declarations for that page which use both the directly defined custom properties and the calculated ones

A nice feature in Google Fonts is that you can select multiple fonts and generate a single embed code to cover them all, so we only need one `@import` statement:

```css
/*
 * Import Desired Google Fonts:
 * - Indie Flower (a handwriting font)
 * - Ubuntu Mono (a monospaced font)
 * - Ubuntu Sans (a sans-serif font)
 */
@import url('https://fonts.googleapis.com/css2?family=Indie+Flower&family=Ubuntu+Mono:ital,wght@0,400;0,700;1,400;1,700&family=Ubuntu+Sans:ital,wght@0,100..800;1,100..800&display=swap');
```

For the most part the `@property` definitions are by-the-book, assigning an initial value, but the computed custom properties are a little different, their definitions contain only `syntax` and `inherits` properties, for example, the accent background colour is simply defined as:

```css
@property --accent-background-color {
  syntax: "<color>";
  inherits: true;
  /* no initial value specified - will be calculated from the accent color */
}
```

The reason for this is that functions can't be used within `initial-value` options in `@property` declarations. This complicates the style sheet a little, because we have to calculate a default value and apply it to the root element later in the style sheet. For the accent background colour that looks like:

```css
/* Calculate default values for the calculated custom properties */
:root {
  /* calculate the accent background color from the accent color */
  --accent-background-color: color-mix(in srgb, var(--accent-color) 10%, white);

  /* … */
}
```

### Notable Custom Property `syntax` Properties

To help illustrate the power of the  `syntax` option in `@property` declarations I intentionally chose the aspect of the page to make customisable to be wide-ranging. The end result is that scanning through the `@property` definitions near the top of the style sheet will show a nice array of different `syntax` options.

The first thing to note is that there is a wide variety of types used, specifically `<string>`  for font names, `<integer>` for pixel and REM counts, `<color>` for colours, `<length>` for sizes in any unit, used for border width and corner radii, and some CSS keywords for the border style. Notice I did not fall back to the generic `*` type, which I simply consider bad practice!

As well as a nice selection of types, I also made sure to use one example of each combinator, so there are comma-delimited lists, space delimited lists, and a collection of allowed special values.

Let's look at some of the most illustrative examples.

Firstly we have a comma-separated list of strings for the default body font family:

```css
@property --body-font {
  syntax: "<string>#";
  inherits: true;
  initial-value: "Ubuntu Sans", "sans-serif"; /* the quotation marks are not optional*/
}
```

We have an integer for the base font size in pixels — note that there is no mechanism for defining valid ranges in CSS, that seems a likely opportunity for future enhancements, but for now, the most explicit we can be is that the pixel size must be a whole number:

```css
@property --base-font-size-px { /* the base font size to use expressed in pixels */
  syntax: "<integer>";
  inherits: true;
  initial-value: 16;
}
```

Next we have a space-delimited list of lengths in any unit for specifying the thickness of the four borders around quotations. Again, there is no mechanism for limiting the number of items to the desired one or four, the best we can do it force it to be list of  space-delimited lengths:

```css
@property --quote-border-width { /* the widge of all borders as a single value, or the top, right, bottom & left width */
  syntax: "<length>+"; /* one or four space-separated widths */
  inherits: true;
  initial-value: 0px 0px 0px 5px;
}
```

And finally, we have an example of a list of allowed CSS keywords for the border style for quotations:

```css
@property --quote-border-style {
  syntax: "dotted|dashed|solid|double";
  inherits: true;
  initial-value: double;
}
```

### Notable Style Definitions

Most of the stye definitions are pretty by-the-book, but one interesting technique I want to draw your attention to is the use of `calc()` for unit conversation.

The best example of this is the spacing around the quotation. The margin and padding are both driven from the `--space-rem` custom property, but that property is an integer, not a REM value. To convert it to a REM value we simply multiply the integer value by an appropriate amount of REM:

```css
blockquote { /* add extra spacing around quotations */
  padding: calc(var(--space-rem) * 1rem); /* needed with background colours and/or borders to move the text in from the edges */
  margin-left: calc(var(--space-rem) * 2rem);
  margin-right: calc(var(--space-rem) * 2rem);
}
```

### Nesting a Custom Property

to illustrate how to set different values for the same custom property on different parts of a page I chose to override the default `--body-font` on block quotes. 

Updating the variable's value on all block quotes and all tags within all block quotes is as easy as:

```css
blockquote {
  /* apply local changes to any needed custom properties before referencing them */
  --body-font: "Indie Flower", "cursive";

  /* … */
}
```

As explained previously, simply updating the value of the custom property has no visible effect, we need to use that value in a style declaration that has a high enough specificity to 'win' over the less specific declaration that has already set the font family to the default value of the `--body-font` custom property.

These two things can be done within a single css statement, if, and only if, the custom
property is updated **before** it's new value is used:

```css
blockquote {
  /* apply local changes to any needed custom properties before referencing them */
  --body-font: "Indie Flower", "cursive";

  /* set the styles, using the localised custom properties */
  font-family: var(--body-font); /* add a more specific font-family declaration to 'win' against the one on the body */
  /* … */
}
```

## Tips & Resources

When developing with CSS custom properties it's useful to notice that the developer tools in modern browsers show custom property values like they do any other CSS property value, but they may collect them together under a separate heading, and they may label them somewhat inaccurately as *'Variables'*.

In **Safari**, you can see the value of all defined custom properties by opening the inspector and clicking on an element. In the styles area, select the *Computer* tab and notice that underneath the diagram of the element's box model is a collapsible list with the values for all the regular CSS properties on the element, and below that is a collapsed list with the values for all the defined *Variables*.

In **Firefox**, similar to Safari, you can select an element in the *Inspector* tab and then you can see both the standard and custom css properties unified into a single list in the *Computed* tab.

At first glance it appears **Google Chrome** doesn't show the custom properties at all, but that's not quite correct. The custom properties are not displayed anywhere in the *Elements* tab by default, but they can be enabled by checking the *Show all* tickbox in the *Computed* tab's filter bar located just under the box model diagram. **Microsoft Edge**, and presumable all the other **Chromium** browsers, behave the same as Google Chrome.

Personally, I find the Safari Developer tools to be the most powerful and easy to use, so I choose to do my development and testing there.

> **For the Curious** — if you're wondering how the JavaScript hangs together, I opted for a very simple design with fully encapsulated code, so absolutely none of the code exists in the global namespace.
>
> To make the code as readable as possible the document ready hander starts by capturing jQuery objects representing the two most important DOM elements in local variables:
>
> ```javascript
> // define variables
> $calculator = $('#calculator');
> $display = $('#calculator input[type="text"].display');
> ```
>
> The code makes heavy use of HTML data attributes to capture metadata right in the HTML. The containing table has a data attribute named `state` initialised to `empty` in the HTML with `data-state="empty"`. jQuery reads and writes data attributes with its `.data()` function, so this saves the need for creating variables. The calculator is a simple state machine that starts in the `empty` state, then moves to the `accumulating` state as the user punches in numbers and operators, then to the `result` or `error` state when the user presses the calculate button (labeled as =).
>
> Because the three types of buttons each have their own class it's easy to attach event handlers to the needed buttons. 
>
> There is just one event handler for all the digits (including the decimal point). So the handler knows which digit was pressed, each number button has a data attribute named `digit` that holds the digit it represents.
>
> There is a similar handler for the operator buttons, and they have a similar data attribute named `operator` that stores the JavaScript arithmetic operator that matches the button's label.
>
> Each of the three action buttons have their own event hander — the clear button simply changes the state to `empty`, the `backspace` handler deletes the last character in the display, and the calculate button calls JavaScript's `eval()` function with the current input inside a `try` block, and writes the appropriate output to the display and moves to the `result` or `error` state depending on whether or not the `eval()` call threw an error.
>
> The simplest of the handlers is literally a one-liner for the clear button, and the most complex of the handers is the one to do the calculation, though even that one is only about twenty lines. The handler for the digits is about average, and a nice illustration of how they all work:
>
> ```javascript
> // add click handlers to the digit buttons
> $('button.digit', $calculator).click(function() {
>   // if the calculator is in the accumulating state, add the digit to the display, otherwise, enter the accumulating state and set the display to the digit
>   if (state() === 'accumulating') {
>     $display.val($display.val() + $(this).data('digit'));
>   } else {
>     state('accumulating');
>     if($(this).data('digit') === '.') {
>       // if the user clicks the decimal point when not accumulating, start with 0.
>       $display.val('0.');
>     }else{
>       $display.val($(this).data('digit'));
>     }
>   }
> });
> ```
>
> The other handlers are a little more complex, but none are more than about 15 lines long, and that's with some extra logic added to handle edge cases for smoothing out the user experience. These edge cases minimise the errors are possible and makes the calculator similarly to a physical pocket calculator. For example, starting a calculation with an operator will pre-fix the operator with a `0`, and pressing any digit or operator when a result is displayed will clear the calculator and start a new calculation.
>
> All the event handlers are implemented as anonymous functions, and to keep the handers simple there are a pair of little helper functions implement as anonymous functions saved to local variables; one to quickly access or update the calculator's state, and one to empty the calculator and start a new calculation:
>
> ```javascript
> // define helper functions
> state = function(newState) {
> if (newState !== undefined) {
>  $calculator.data('state', newState);
> }
> return $calculator.data('state');
> };
> empty = function() {
> state('empty');
> $display.val('0');
> };
> ```
